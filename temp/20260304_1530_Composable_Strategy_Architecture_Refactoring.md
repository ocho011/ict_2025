## 1. 아키텍처 위계 및 용어 정의 (Architectural Hierarchy)
본 프로젝트에서 명명 규칙(Naming)은 단순한 이름이 아니라 시스템 내의 **위계와 책임**을 정의합니다.

*   **전략 (Strategy)**: 최상위 **컨테이너(Container)이자 오케스트레이터(Orchestrator)**입니다.
    *   *역할*: 데이터 스트림을 관리하고, 여러 결정자(Determiners)를 조율하여 최종 매매 신호를 발행하는 **'파이프라인'** 역할을 수행합니다.
    *   *지향점*: 로직을 직접 소유하지 않고 엔진과의 통신 및 부품 간 데이터 흐름(Context Flow)에만 집중하는 **'Lean Interface'**를 지향합니다.
*   **결정자 (Determiner / Module)**: 전략 내부에서 특정 기능만 수행하는 **독립된 로직 유닛(Logic Unit)**입니다.
    *   *역할*: Entry, Exit, StopLoss, TakeProfit 등 개별적인 의사결정만 담당하며, 엔진이나 지갑 상태를 몰라도 작동하는 **'Pure Logic'**을 지향합니다. (예: `EntryDeterminer`, `ExitModule`)

## 2. 현재 상태 및 과도기적 문제점 (Transition Issues)
현재 시스템은 모노리식(Monolithic) 구조에서 조립식(Composable) 구조로 전환되는 **'하이브리드 과도기'** 상태입니다.

*   **진입(Entry)의 경직성**: 청산(Exit)은 이미 이름 기반 조립식인 반면, 진입 로직은 여전히 `strategy_params` 섹션에 특정 전략(ICT) 전용 파라미터가 고정된 형태로 나열되어 있습니다.
*   **디렉토리 파편화**: `detectors`, `entry`, `exit`, `pricing`, `strategies` 폴더가 최상위에 산재하여 부품(Determiner)과 엔진(Strategy)의 구분이 모호합니다.
*   **비대한 인터페이스 (Fat Interface)**: 기존 `Strategy` 추상 클래스가 로직(내용물)과 엔진(프레임워크)을 혼재시켜 동적 조립을 저해하고 있습니다.

## 3. 리팩토링 핵심 목표 (Core Goals)
1.  **Full Composable (완전 조립식)**: 모든 진입/청산 로직을 이름 기반으로 모듈을 선택하고 파라미터를 동적으로 주입하는 방식으로 통일.
2.  **Structural Integrity (구조화)**: 모든 로직 부품을 `src/strategies/modules/` 하위로 통합하여 '부품 창고'를 일원화.
3.  **Naming Alignment (명칭 통일)**: 로직 유닛의 명칭에서 'Strategy'를 제거하고 `Determiner` 또는 `Module` 접미사로 통일.

### [설계 변경 예시: Fixed → Composable]
*   **AS-IS (현재)**:
    ```yaml
    strategy_params:
      active_profile: "RELAXED" # ICT 전용 필드 고정
    ```
*   **TO-BE (목표)**:
    ```yaml
    entry_config:
      entry_strategy: "ict_optimal_entry" # 부품 이름 선택
      parameters:
        active_profile: "RELAXED" # 해당 부품 전용 파라미터 주입
    ```

## 4. 시스템 5단계 공정 및 인스턴스 생성 경로
설계도(YAML)가 실제 심볼별 전략 객체로 탄생하는 정밀 공정입니다. (예: `BTCUSDT`)

*   **Step 1. Module Definition**: 개발자가 `@register_module`을 사용하여 부품을 등록하고 `ParamSchema`로 규격을 정의합니다.
*   **Step 2. Registry Management**: `ModuleRegistry`가 모든 부품 클래스를 메모리에 로드하여 중앙 창고를 구축합니다.
*   **Step 3. Blueprint & Merge (ConfigManager)**: `base.yaml`의 `defaults`와 `symbols['BTCUSDT']`의 오버라이드 설정을 병합하여 최종 심볼 설정을 확정합니다.
*   **Step 4. Dynamic Assembly (DynamicAssembler)**: YAML에 적힌 이름으로 클래스를 조회하고, 주입된 파라미터의 유효성을 검증하여 4개의 독립된 **결정자(Determiner)** 인스턴스를 생성합니다.
*   **Step 5. Orchestration & Execution**: 생성된 결정자들을 `ComposableStrategy` 컨테이너에 주입하고, `TradingEngine`이 이를 실시간 데이터 스트림과 연결하여 가동합니다.

## 5. 확장된 리팩토링 범위 (Extended Scope)
단순한 구조 개선을 넘어 시스템의 성능과 안정성을 위해 다음 사항을 병행합니다.

*   **Non-blocking IO**: 동기 방식의 바이낸스 API 호출을 비동기(`asyncio`) 구조로 전환하여 엔진 지연시간 최소화.
*   **Full Dependency Injection**: 싱글톤 패턴을 지양하고 모든 인프라 객체를 Assembler를 통해 주입받아 테스트 용이성 확보.
*   **Centralized Indicator Store**: 중복 지표 계산을 방지하기 위해 결정자들이 공유하는 전용 **'Feature Store'** 도입.
*   **Unified Resilience Policy**: 서킷 브레이커와 재시도 로직을 정책 모듈로 분리하여 통합 관리.
