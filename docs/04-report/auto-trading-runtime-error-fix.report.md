## 1. Executive Summary
바이낸스 무기한 선물 자동매매 운영 중 발생한 `Binance API error: {'code': -4130, 'msg': 'An open stop or take profit order with GTE and closePosition in the direction is existing.'}` 에러를 분석하고 해결하였습니다. 

본 에러는 Trailing Stop 로직이 Stop Loss(SL)를 업데이트할 때, 기존에 존재하던 익절(TP) 주문과의 `closePosition=true` 옵션 충돌을 제대로 처리하지 못해 발생한 것으로 확인되었습니다. 필터링 로직을 확장하고 동기화(Sync) 프로세스를 강화하여 문제를 해결하였으며, 단위 테스트를 통해 검증을 완료했습니다.

## 2. Root Cause Analysis
*   **바이낸스 제약 사항:** 바이낸스 선물 API는 동일한 방향(Side)에 대해 `closePosition=true` 옵션이 설정된 알고리즘 주문(SL 또는 TP)을 단 하나만 허용합니다.
*   **기존 로직의 결함:** `update_stop_loss` 함수가 새로운 SL을 넣기 전 기존 주문을 취소할 때, 오직 `STOP` 및 `STOP_MARKET` 타입만 조회하여 취소 대상에 포함했습니다.
*   **충돌 시나리오:** 만약 익절 주문(`TAKE_PROFIT_MARKET`)이 `closePosition=true`로 먼저 걸려 있다면, `update_stop_loss`는 이를 인지하지 못하고 취소하지 않은 채 새로운 SL 주문을 시도하게 되어 -4130 에러가 발생했습니다.
*   **재시도 로직 한계:** 에러 발생 시 시도되는 `force re-syncing` 과정에서도 TP 타입을 누락하여 반복적인 실패가 발생했습니다.

### 3.1 Code Modifications
`src/execution/order_gateway.py` 파일의 `update_stop_loss` 메서드를 다음과 같이 수정했습니다.

## 4. Risks and Mitigation (Act)
*   **리스크:** TP 주문이 SL 업데이트 과정에서 일시적으로 사라졌다가 다시 생성되지 않을 경우(SL만 남을 경우).
*   **대응:** 현재 시스템 설계상 `update_stop_loss`는 SL 관리 전용이며, TP는 별도의 로직으로 관리됩니다. 향후 SL과 TP를 하나의 'Exit Strategy' 모듈로 통합 관리하여 두 주문의 공존 상태를 원자적으로(Atomic) 관리하는 리팩토링을 권장합니다.

## 5. Lessons Learned
*   바이낸스 API의 특정 옵션(`closePosition`)이 가지는 배타적 제약 조건을 코드 레벨에서 더 엄격하게 반영해야 함을 확인했습니다.
