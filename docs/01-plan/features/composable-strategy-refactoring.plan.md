## Context
- **Target Project:** Composable Strategy-based Cryptocurrency Automated Trading System
- **Reference Document:** 20260304_1530_Composable_Strategy_Architecture_Refactoring.md (Includes Architectural Hierarchy, Naming Conventions, 5-Step Instance Creation Process, Extended Refactoring Scope)
- **Purpose:** Transition from Monolithic/Hybrid structure to Full Composable structure and execute PDCA cycle until reaching an executable state (verifying actual position entry).

## Goal
- **Unified Logic:** Standardize all Entry/Exit/SL/TP logic to a name-based module selection + parameter injection method.
- **Slim Strategy:** Slim down Strategy to a pure Container/Orchestrator (Lean Interface), concentrating business logic in Determiner/Module.
- **Directory Structure:** Unify module directories to `src/strategies/modules/` and align suffixes to `Determiner` or `Module`.
- **Process Stabilization:** Refine `ModuleRegistry`, `ConfigManager`, and `DynamicAssembler` to stabilize the 5-step process: `base.yaml` → `symbol config` → `Determiner instance` → `ComposableStrategy container`.
- **Performance & Reliability:** Introduce Non-blocking IO (asyncio), Full DI, Feature Store, and Resilience Policy to improve performance and reliability.
- **Execution Verification:** Ensure trading and position entry occur normally in actual execution tests, and trace/record causes in case of failure.

## Scope
- **Code Level:** Strategy Abstract Class, ComposableStrategy, Determiner/Module Implementation, Module Registration/Registry, ConfigManager, DynamicAssembler, TradingEngine Interface, Binance Client, Feature Store, Resilience Policy.
- **Configuration Level:** Normalize schemas for `base.yaml`, symbol-specific YAMLs, and `entry_config`/`exit_config`/`stop_loss_config`/`take_profit_config`.
- **Test Level:** Unit Tests (Determiner), Integration Tests (Config→Assembler→Strategy→Engine→Mock Client), Real or Sandbox Dry-run Tests.

## Work Breakdown (Parallel Agent Units)

### 1. Architecture & Interface Refactoring Agent
- Remove business logic from Strategy abstract class, leaving only Container/Orchestrator roles.
- Redesign `ComposableStrategy` interface to receive Entry/Exit/SL/TP Determiner instances and route signals.
- Clearly define boundaries between `TradingEngine` and `Strategy`, and specify signal/order/state flows.

### 2. YAML & Config/Assembler Schema Agent
- Remove or migrate existing `strategy_params` based ICT-specific fields.
- Define schemas for `entry_config`/`exit_config`/`stop_loss_config`/`take_profit_config`.
  - Introduce `strategy` (module name) and `parameters` blocks in each section.
- Redefine merge rules for `base.yaml` and `symbols[*]`, and structure `ConfigManager` to perform primary validation.

### 3. Module Renaming & Directory Reorganization Agent
- Integrate scattered logic (detectors, entry, exit, pricing, strategies) under `src/strategies/modules/`.
- Remove 'Strategy' from names and unify suffixes to `Determiner` or `Module` (e.g., `XxxEntryStrategy` → `XxxEntryDeterminer`).
- Apply `@register_module` and `ParamSchema` to enable central management by `ModuleRegistry`.

### 4. ModuleRegistry / DynamicAssembler Refinement Agent
- Implement/Refine `@register_module` decorator and strengthen parameter validation logic based on `ParamSchema`.
- Implement `DynamicAssembler` as a single responsibility class.
  - Create Entry/Exit/SL/TP Determiner instances taking YAML configuration as input.
  - Output clear exceptions/logs upon module name mapping failure or parameter validation failure.

### 5. Infrastructure (Async, DI, Feature Store, Resilience) Agent
- Switch Binance API calls to asyncio-based asynchronous, propagating `await` throughout the call path.
- Change structure to inject Engine, Client, Repository, Feature Store via Assembler or DI Container.
- Move common indicator calculation logic to Feature Store, allowing Determiners to query values only via Feature Store.
- Modularize Retry/Circuit Breaker policies and inject them into HTTP/WS Clients.

### 6. Test & Execution Verification Agent
- **Determiner Unit Test:** Verify correct output of LONG/SHORT/FLAT signals based on input data.
- **Integration Test:** Implement flow Config→ConfigManager→DynamicAssembler→ComposableStrategy→TradingEngine→Mock Binance Client.
- **Dry-run Test:** Verify the entire chain from Signal → Order → Position Entry in Real or Sandbox environment.
- **Failure Analysis:** Write checklists and automated assertions based on past "position not entering" failure cases.

## Risks & Assumptions
- **Risk:** Dependency on legacy code may be deeper than expected, requiring extensive modification during interface changes.
- **Risk:** Async transition may introduce subtle concurrency/race conditions.
- **Risk:** YAML schema changes may conflict with existing configuration files, breaking compatibility with the old system.
- **Assumption:** Business logic (e.g., ICT conditions) remains based on current implementation, with structural relocation being the principle.
- **Assumption:** The design principles in the reference document take precedence as the higher norm; in case of conflict with legacy implementation, the design document is chosen.
